# Data Directory

This directory contains data tracked by [DVC (Data Version Control)](https://dvc.org/). The actual data files are **not** stored in GitHub. Instead, small `.dvc` pointer files are committed to Git, and the real data lives on Box.

---

## Why DVC?

We use DVC to tie data versioning to code versioning. Every Git commit references a specific version of the data through `.dvc` pointer files. This means you can check out any past commit and retrieve the exact data that was used with that version of the code — ensuring full reproducibility.

---

## How It Works

### Architecture

```
GitHub (Git)                        Box (DVC Remote)
├── src/                            └── data/
├── notebooks/                          └── files/md5/
├── reports/                                ├── a1/b2c3d4...
├── data/                                   ├── d7/e8f9a0...
│   ├── raw.dvc  ──── points to ──────────► │
│   ├── processed.dvc ── points to ───────► │
│   ├── .gitignore                          └── ...
│   └── README.md
```

- **Git** tracks code, notebooks, reports, and `.dvc` pointer files.
- **Box** stores the actual data in a hashed format managed by DVC.
- The `.dvc` pointer files are the bridge — they record which data hash belongs to which Git commit.

### What DVC Does to Your Data

When you run `dvc add` on a data directory:

1. DVC calculates a unique hash (fingerprint) for each file based on its contents.
2. The files are moved into a local cache (`.dvc/cache/`).
3. A small `.dvc` pointer file is created, storing the hash.

When you run `dvc push`:

- The cached files are copied to the Box remote, stored by their hash (not by filename).
- Old versions are never overwritten — each version coexists alongside others.

When you run `dvc pull`:

- DVC reads the `.dvc` pointer files, finds the matching hashes on Box, and restores the original files with their original names and structure.

## Directory Structure

```
data/
├── raw/            # Original, unmodified data (tracked by DVC)
├── processed/      # Cleaned and transformed data (tracked by DVC)
├── raw.dvc         # DVC pointer file for raw/ (tracked by Git)
├── processed.dvc   # DVC pointer file for processed/ (tracked by Git)
├── .gitignore      # Auto-generated by DVC to exclude raw/ and processed/ from Git
└── README.md       # This file
```

## Daily Workflow

### Starting a Work Session

If the data is not on your machine (or you want to make sure it's up to date):

```bash
dvc pull
```

This restores `data/raw/` and `data/processed/` with the original filenames. You can then work with them normally in your scripts and notebooks:

```python
import pandas as pd

df = pd.read_csv("data/raw/your_file.csv")
```

### After Modifying Data

If you make changes to the data:

```bash
# 1. Track the changes with DVC
dvc add data/raw
dvc add data/processed

# 2. Push updated data to Box
dvc push

# 3. Commit the updated pointer files to Git
git add data/raw.dvc data/processed.dvc
git commit -m "updated raw/processed data"
git push
```

### When You Haven't Changed the Data

Do nothing. The local copy and remote stay in sync. No need to push or pull.

### Freeing Up Disk Space (Optional)

If you need to reclaim space on your machine, you can delete the local data:

```bash
rm -rf data/raw data/processed
```

The `.dvc` pointer files remain, and the data is safe on Box. Run `dvc pull` whenever you need it back.

## Switching Between Versions

### Finding Past Versions

First, find the commit you want to go back to:

```bash
# View commit history
git log --oneline

# Example output:
# f3a1b2c  updated raw data with new patient records
# a7d9e1f  initial commit
```

### Scenario 1: Temporarily Viewing Old Data

If you just want to look at or test against an older version of the data, you do **not** need to commit anything:

```bash
# Check out the old commit (this puts you in "detached HEAD" state)
git checkout <commit-hash>

# Pull the matching data from Box
dvc pull
```

You now have the exact data that existed at that point in time. You can explore it, run old notebooks against it, etc.

When you're done, return to your current work:

```bash
# Go back to the latest version
git checkout main

# Restore the current data
dvc pull
```

No commits are needed for this — you're just browsing history.

### Scenario 2: Permanently Reverting to Old Data

If you want to go back to an old version of the data **and continue working from there**, you need to create a new branch:

```bash
# Check out the old commit
git checkout <commit-hash>

# Create a new branch from this point
git checkout -b reverted-data

# Pull the old data
dvc pull
```

Now you're on a new branch with the old data. You can continue working from here. If you make changes, follow the normal workflow (`dvc add`, `dvc push`, `git add`, `git commit`, `git push`).

### Scenario 3: Restoring Old Data on Your Current Branch

If you want to bring back old data without leaving your current branch:

```bash
# Restore just the .dvc pointer files from an old commit
git checkout <commit-hash> -- data/raw.dvc data/processed.dvc

# Pull the old data from Box
dvc pull

# Commit the restored pointer files
dvc push
git add data/raw.dvc data/processed.dvc
git commit -m "reverted data to version from <commit-hash>"
git push
```

This keeps you on `main` (or whatever branch you're on) but swaps the data back to an older version. A new commit is required here because you're changing what version of the data your current code points to.

### Important Notes on Version Switching

- **Always make sure your current work is committed before switching versions.** If you have uncommitted changes to code or `.dvc` files, switching can cause conflicts.
- **`dvc pull` must be run after every `git checkout`** that changes the `.dvc` pointer files. Git updates the pointers, but DVC needs to fetch the matching data.
- **If you get a warning about uncommitted changes**, either commit them first or stash them with `git stash` (and `git stash pop` to restore later).
- **`dvc status`** is your friend — run it anytime to check if your local data matches what the `.dvc` pointer files expect.

## Setting Up a New Project

For reference, here is how to set up DVC for a new project following this same pattern:

```bash
# Create project locally
mkdir new-project
cd new-project
mkdir src notebooks reports data

# Initialize Git and DVC
git init
dvc init

# Set DVC remote to a Box folder
dvc remote add box ~/Library/CloudStorage/Box-Box/Projects/new-project/data
dvc remote default box

# Add data, push, and commit
dvc add data/
dvc push
git add .
git commit -m "initial commit"
git remote add origin <github-repo-url>
git push
```

## Key Commands Reference

| Command | What It Does |
|---|---|
| `dvc pull` | Download data from Box to your local machine |
| `dvc push` | Upload local data to Box |
| `dvc add data/raw` | Track changes to `data/raw/` and update the pointer file |
| `dvc remote list` | Show configured remotes |
| `dvc status` | Check if local data is in sync with `.dvc` pointer files |
| `dvc diff` | Show what data changed between commits |

## Important Notes

- **You must have the data locally to work with it.** There is no way to read directly from the hashed format on Box. Always `dvc pull` before working.
- **Box Drive must be installed and synced** on any machine that needs to push or pull data.
- **Do not manually edit files in the Box remote folder.** DVC manages that structure — editing it directly will break things.
- **Always commit `.dvc` files to Git after pushing data.** If you `dvc push` but forget to `git commit` and `git push` the pointer files, your code and data versions will be out of sync.
